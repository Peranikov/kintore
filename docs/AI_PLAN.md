# AIプラン決定ロジック

`/plan-create` で使う AI プラン生成の判断材料と優先順位をまとめる。
実装の詳細は `src/services/gemini.ts` を基準にし、この文書では意思決定ルールを整理する。

## 目的

- 筋トレ効果を出しやすい部位と種目を AI が選びやすくする
- 完全な固定メニュー化を避けつつ、再現性のある提案を出す
- ログが増えても、なぜその部位・種目を選んだかを追いやすくする

## 入力情報

AIプラン生成では以下の情報を使う。

- 構造化プロフィール
  - 主な目標
  - トレーニング歴
  - 週あたりの頻度
  - 1回あたりの時間
  - 強化したい部位
  - 部位ごとの目標週セット数
  - 痛み・避けたい動き
  - 体格・体組成メモ
  - 補足メモ
- 登録済み種目
  - 種目名
  - 部位
  - カテゴリ
  - 自重/有酸素フラグ
- 直近7回分のログ
- 直近の保存済み AI評価
- 停滞検出結果
- ディロード提案結果
- 当日のユーザー入力メモ

## 部位優先の決め方

部位優先は `胸 / 背中 / 肩 / 脚 / 腕 / 体幹` を対象に判定する。
`有酸素` と `その他` は優先部位の算出対象から外す。

### 1. 週セット数の集計

- 基準日は「今日」ではなく、直近ログの最新日
- その最新日から過去7日間のセット数を部位別に集計する
- 目標週セット数が設定されている場合は `実績 / 目標` として扱う

### 2. 前回実施日の記録

- 各部位について、最後に実施したログ日を保存する
- 同率比較では、前回実施から日数が空いている部位を優先する

### 3. 優先順位

優先順位は次の順に決める。

1. 目標週セット数に対する不足量が大きい部位
2. 不足量が同じ場合は、直近1週間のセット数が少ない部位
3. さらに同じ場合は、前回実施から日数が空いている部位
4. 最後は部位名順

## 今日の推奨部位

- 優先順位1位を `今日の推奨部位` とする
- 理由として以下をプロンプトに含める
  - 目標に対する不足量、または直近1週間のセット数不足
  - 前回実施日と経過日数

AI には、ユーザーの当日メモで明示的な希望がない限り、この推奨部位を最優先でプランの軸にするよう指示する。

## 優先候補部位

- 上位3部位を `今日の優先候補部位` としてプロンプトへ渡す
- 推奨部位以外の候補は補助候補として扱う
- ユーザーの当日メモで希望部位がある場合は、その希望を優先する

## 推奨種目候補の決め方

推奨部位に属する登録済み種目から候補を作る。

### 含める情報

各候補種目に以下を付ける。

- カテゴリ
- 自重フラグ
- 前回実施日
- 前回セット内容
- 停滞中フラグ
- 刺激変化候補フラグ
- 連続採用中フラグ

### 並び順

候補種目は次の順に並べる。

1. 停滞中の種目
2. 刺激変化候補
3. 連続採用回数が少ない種目
4. 最近やっていない種目
5. 名前順

## 刺激変化候補

推奨部位で停滞中の種目がある場合、以下を行う。

- その部位で直近最後に使ったカテゴリを確認する
- それと異なるカテゴリの種目を `刺激変化候補` とする
- AI には、停滞打破のために補助種目で別カテゴリを1つ混ぜるよう指示する

## 連続採用中

固定化を避けるため、同一種目が連続採用中かを判定する。

- ログを新しい順に見て、同じ種目が何セッション連続で含まれているかを数える
- 2回以上続いている場合は `連続採用中` として候補に表示する
- AI には、主軸種目は維持してもよいが、補助種目では別候補を混ぜるよう指示する

## 停滞・ディロードとの関係

- 停滞中の種目は、部位優先と候補種目の両方に影響する
- ディロード推奨がある場合は、部位優先より回復優先の制約が強い
- したがって、ディロード時はボリュームを 50-60% に抑える指示が優先される

## 今後の拡張候補

- 種目ごとの推奨レップ帯
- 種目ごとの負荷進行ルール
- 部位ごとの最小/最大推奨セット数
- 目的別テンプレート（筋肥大 / 筋力 / 減量）
- 当日の疲労・睡眠・痛みを使った推奨部位補正
